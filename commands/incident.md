# incident — 장애 분석 리포트 생성

장애 보고를 받아 구조화된 분석 리포트를 생성한다.
코드 수정은 하지 않는다. 리포트만 산출.

## 입력

`$ARGUMENTS`에서 장애 소스를 파싱한다:

| 패턴 | 소스 타입 | 도구 |
|------|----------|------|
| Jira URL 또는 티켓 ID | Jira 이슈 | `mcp__jira__jira_get_issue` |
| Slack 메시지 URL | Slack 스레드 | `mcp__slack__get_thread_replies` |
| 텍스트 | 구두 보고 | 그대로 구조화 |

인자 없으면 장애 내용 요청.

## 실행 (5단계 교차 검증)

<steps>

### 1단계: 현상 파악 + 재현

- 장애 보고 내용 정리 (누가, 언제, 어떤 현상)
- 재현 조건 특정 (환경, 입력, 순서)
- 영향 범위 파악 (영향 유저 수, 기능 범위, 데이터 오염 여부)
- 롤백 가능 여부 확인

### 2단계: 가설 수립 (최소 3개)

코드베이스를 분석하여 원인 가설 3개 이상 수립.
각 가설은 아래 형식으로:

```
가설 A: {원인 설명}
- 근거: {코드 위치, 로그, 조건}
- 반증: {이 가설이 틀리다면 관찰돼야 할 것}
```

가설이 1-2개만 떠오르면 억지로 채우지 않되, "다른 원인 가능성을 충분히 탐색했는가?" 자문.

### 3단계: 증거 대조

각 가설별로:
- **증거** (이 가설을 지지하는 코드/로그/데이터)
- **반증** (이 가설에 반하는 사실)
- **확신도** (높음/중간/낮음)

증거 없이 확신도 "높음" 판정 금지.

### 4단계: 수정 방향 + 사이드이펙트

가장 유력한 가설 기준으로:
- 수정 방향 (구체적 파일/함수 수준)
- 의존 경로 분석 (이 코드를 호출하는 곳 전부 탐색)
- 사이드이펙트 (수정 시 깨질 수 있는 경로)
- 유사 패턴 (같은 버그가 코드베이스 다른 곳에도 있는지)

### 5단계: 반대 심문 (자기 반박)

자신의 분석을 의도적으로 공격:
- "이 분석이 틀렸다면 어디서 틀렸을까?"
- "선택한 가설 외에, 증거가 설명하는 다른 원인은?"
- "수정 방향이 실제 원인을 해소하는가, 증상만 가리는가?"
- "사이드이펙트 분석에서 빠뜨린 경로는?"

반대 심문 결과 확신도가 낮아지면 2단계로 돌아가 가설 재수립.

</steps>

## 예시

<example>
입력: `/incident PROJ-789 결재 승인 버튼 클릭 시 500 에러`

산출물 일부:
```markdown
## 4. Root Cause Analysis

### Hypothesis A: approval status enum 불일치 ⭐ (유력)
- 원인: DB enum 'APPROVED'와 코드 enum 'Approved' 대소문자 불일치
- 증거: ApprovalService.java:142 — status.equals("APPROVED") vs DB column CHECK ('Approved')
- 반증: 없음
- 확신도: 높음

### Hypothesis B: 동시성 문제 (Race condition)
- 원인: 두 결재자가 동시에 승인 시 lock 없음
- 증거: ApprovalRepository.java에 @Lock 미사용
- 반증: 에러 로그에 concurrent 관련 로그 없음. 단일 유저 재현 가능.
- 확신도: 낮음
```
</example>

<example>
입력: `/incident 프론트에서 알림 설정 저장 안 됨`

산출물 일부:
```markdown
## 10. Self-Review (반대 심문 결과)
- 분석 약점: 네트워크 탭 로그 미확인 — API 응답은 200인데 실제 저장 안 되는 경우 가능
- 미확인 사항: staging 환경에서 DB 직접 확인 필요
- 최종 확신도: 중간
```
</example>

## 완료 후

리포트 생성 완료 시:

1. `docs/incident-report.md` 저장
2. 사용자에게 리포트 요약 출력
3. 다음 단계 안내:
   - **적대적 리뷰**: `/codex:rescue` 로 리포트의 분석 정확성 교차 검증
   - **수정 구현**: 리포트 기반으로 별도 작업 (이 스킬 범위 밖)

## 규칙

- 코드 수정 절대 금지. 분석 리포트만 산출.
- 확인 불가한 것은 `[미확인: <이유>]` 표기. 추측을 사실처럼 쓰지 않음.
- 가설은 "유력" 표시 하나만. 동률이면 둘 다 유력 표기하되 추가 확인 필요 명시.
- 반대 심문(5단계) 건너뛰기 금지 — 확신도 높아도 반드시 실행.
- Evidence는 `파일:라인` 형식. 코드 인라인 출력은 핵심 1-2줄만.
