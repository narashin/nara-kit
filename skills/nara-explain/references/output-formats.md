# Output Format Templates

## Tech Peer (팀원 / 리뷰어)

구현 세부사항과 설계 결정 근거를 포함한다.

```markdown
## [기능/변경 이름]

### 한 줄 요약
[무엇을, 왜]

### 배경
[왜 이 변경이 필요했는지]

### 구현 접근
[어떻게 구현했는지]
- 선택한 방식: [이유]
- 고려했던 대안: [왜 선택 안 했는지]

### 핵심 변경 파일
- `path/to/file.ts` — [역할]
- `path/to/other.ts` — [역할]

### 주의사항 / 사이드이펙트
[있을 경우만]
```

---

## New Member (신규 입사자 / 온보딩)

전체 맥락과 진입점 중심. 핵심 폴더만, 전체 나열 금지.

```markdown
## [프로젝트/기능 이름] 온보딩 가이드

### 무슨 프로젝트인가요?
[비기술적 한 줄 설명]

### 기술 스택
- [스택 1]: [한 줄 역할]
- [스택 2]: [한 줄 역할]

### 디렉토리 지도
```
[핵심 폴더만]
```

### 진입점
1. `path/to/entry.ts` — 여기서 시작
2. → `path/to/next.ts` — 다음으로 이어짐

### 자주 작업하는 곳
- [작업 유형]: `path/`

### 로컬 실행
```bash
[핵심 커맨드만]
```
```

## Non-tech / PM (기획자 / 비개발자)

코드 스니펫 포함 금지. 사용자 시나리오와 영향 중심.

```markdown
## [기능 이름]

### 무슨 기능인가요?
[비유 또는 사용자 시나리오로 설명]

### 왜 필요한가요?
[문제 → 해결]

### 사용자는 어떻게 달라지나요?
- 이전: [기존 경험]
- 이후: [변경된 경험]

### 알아두면 좋은 것
[제약이나 한계가 있을 경우만]
```

---

## External / 발표 (컨퍼런스 / 외부 공유)

청중이 공감할 수 있는 문제부터 시작. 기술 용어 최소화.

```markdown
## [제목]

### 문제
[청중이 공감할 수 있는 문제 서술]

### 접근
[왜 이 방식을 선택했는지]

### 결과
[수치, 개선, 임팩트]

### 핵심 교훈
[1~2개]
```

---

## Brief 형식 (슬랙 / 이메일 / 한 문단)

사용자가 "짧게", "슬랙에 올릴", "한 문단" 요청 시 사용.

```
[한 줄 요약]

- [핵심 포인트 1]
- [핵심 포인트 2]
- [핵심 포인트 3]
[필요 시 한 줄 맥락]
```
